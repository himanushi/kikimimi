# issue の実装

`issues/NNNNN/plan.md` を受け取ってから、コミットして終えるまでの手順。順番を飛ばさない。

## 手順

### 1. 文脈の読み込み

- `issues/NNNNN/plan.md` を読む(受け入れ基準とやらないことを把握)
- `knowhow/` を確認する。M5Stack / PlatformIO 系は前作 `../yesman/knowhow/` の `m5stack-*`, `pio-*` も読む

### 2. TDD(red → green)

- 受け入れ基準からテストを先に書く(実装の写経ではなく仕様から導く)
- **テストが落ちることを確認する(red)**。落ちなければテストが間違っている
- 最小の実装で green にする。green 後に必要ならリファクタ
- ファームウェアでテスト困難な部分: ロジックは純関数に分離してホスト側でテストし、ハードウェア依存部は実機確認(証拠つき)で代替する

### 3. 自己検証

- typecheck / test / ビルドをすべて実行し、**出力を読む**。成功は主張でなく証拠(出力)で示す
- 受け入れ基準を 1 つずつ、plan.md に書いた検証方法で確認してチェックを付ける

### 4. コードレビュー(subagent)

フレッシュコンテキストの subagent に **diff と plan.md(受け入れ基準・やらないこと)だけ**を渡してレビューさせる。実装の経緯・自分の推論は渡さない。レビュー観点:

- **要件適合**: 受け入れ基準を本当に満たすか。テストを通すためのハードコード・特殊分岐(reward hacking)がないか
- **テストの実質**: テストが実装の写経(トートロジー)になっていないか。壊したら落ちるか
- **エラーハンドリング**: 例外を握りつぶして無言で fallback していないか(fail-open)。逆に、起こり得ないシナリオへの防御コードを足していないか
- **スコープ**: plan.md の「やらないこと」を侵していないか。頼まれていないリファクタ・抽象化がないか
- **依存**: 新規パッケージは実在するか、正規名か(公式レジストリで照合)
- **public repo**: 個人情報・API キー・デバイス固有情報が diff に含まれていないか

指摘は正しさ・要件に影響する高信頼のものだけ対応する。スタイル好みの指摘は流す。

### 5. 記録

- `issues/NNNNN/result.md`: 実装報告。次の 3 点を書く:
  1. **検証の証跡**: 受け入れ基準をどう検証したか
  2. **判断と経緯**: 途中で選んだ・変えた方針と「なぜそうしたか」、plan からの逸脱とその理由
  3. **レビュー指摘と対応**
- ハマった点のうち**次の issue でも踏みそうなもの**は `knowhow/<カテゴリ>-<スラッグ>.md` に 1 件 1 ファイルで蒸留する(result.md からはリンクで参照)
- 進行の経過はコミット履歴が記録する。repo が既に記録していることを result.md に書き写さない
- `docs/ROADMAP.md` があれば該当項目を更新

### 6. ADR 昇格判断

result.md の「判断と経緯」を見返して問う: **「この判断は今後の issue でも前提になるか?」**

- Yes(ツール選定・構成・方式など、プロジェクト共通の概念になった)→ `docs/decisions/` に ADR を追記し、result.md から ADR 番号を参照する
- No(この issue 限りの判断)→ result.md に残したままでよい

### 7. コミット

- 日本語メッセージ、prefix `add: / update: / fix: / refactor: / docs:`
- 1 スライス = 1 コミット。**常に動く状態で終える**(ビルドが通らない状態でコミットしない)
- コミットメッセージに個人情報・デバイス固有情報を含めない

## 中断条件

- 検証が 3 回連続で失敗したら手を止め、状況と試したことを正直に報告する
- plan.md の前提が実装中に崩れたら、勝手にスコープを広げず、plan.md の修正を提案する
