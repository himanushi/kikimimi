# issue の作成

タスクを実装単位に切り、`issues/NNNNN/plan.md` を作成する手順。

## 手順

1. **番号の決定**: `issues/` 配下の既存ディレクトリを確認し、次の 5 桁番号を採番する(初回は `00001`)
2. **要求の明確化**: What/Why が曖昧なら、着手前に的を絞った質問をする。特にエッジケース・やらないことの境界を確認する
3. **plan.md の作成**: 下のテンプレートで書く。**受け入れ基準は 1 つずつ検証方法(コマンド・観察手順)を名指しできる粒度**にする
4. **サイズの確認**: 1 issue = 1〜数コミットで終わるサイズに切る。大きければ複数 issue に分割し、依存関係を明記する
5. **承認**: 大きい変更は plan.md をユーザーに見せて承認を得てから実装に進む。diff を一文で説明できる小修正はこの skill 自体が不要

## plan.md テンプレート

```markdown
# NNNNN: <タイトル>

## What / Why

<何を作るか。なぜ必要か(誰の何を可能にするか)>

## 受け入れ基準

- [ ] <基準>(検証: <コマンドまたは観察手順>)

## やらないこと

- <スコープ外を明示>

## How(実装方針)

<触るファイル、アプローチ。依存する issue があれば番号>
```

## 注意

- public リポジトリ。個人情報・API キー・デバイス固有情報を plan.md に書かない
- 触るファイルが交差しない issue は並列実行可能なので、分割時はファイル境界を意識する
