# M5Unified の Touch.wasPressed() は次の M5.update() まで立ちっぱなし — 同一ループ内のモード遷移で二重処理される

loop() 内で画面モードごとに独立した `if (mode == ...)` ブロックを並べてタッチを処理すると、タップでモードが変わった直後、同じループ内の遷移先ブロックが**同じタップ**をもう一度処理する。`wasPressed()` はエッジ検出だが、フラグは次の `M5.update()` 呼び出しまでクリアされないため。

## 症状

kikimimi で「WiFi / API キー設定」をタップすると URL QR 画面が開いた瞬間に閉じる(遷移先の「タップで閉じる」処理が同一タップで発火)。ユーザーには「ボタンを押しても反応しない」ように見える。ボタン座標が遷移先のボタンと重なっている場合は、意図しない操作(即戻る等)にもなる。

## 対策

モード分岐を `else if` で連鎖させ、1 ループ 1 モードだけがタッチを処理するようにする。ブロック内でモードが変わっても、その iteration では後続ブロックが評価されない。

## なぜ重要か

画面(モード)が増えるたびに同じ罠を踏む。タッチ処理のあるモードブロックを追加するときは必ず else-if 連鎖に繋ぐこと。
