# NVS の旧形式キーからリスト形式へ移行するときは、移行元キーを保存時に消す

## 何が起きたか

WiFi 設定を単一 `ssid`/`pass` キーから複数登録リスト(JSON 1 キー)へ移行する実装で、
`configLoad()` 内で「リストに無ければ旧キーの値を追加する」という移行ロジックを組んだ。
これ自体は正しいが、移行後も旧キーを NVS に残したままだったため、
ユーザーが設定ページでその(移行された)WiFi を削除して保存しても、
次回起動時の `configLoad()` が再び旧キーを読み、リストに存在しない(削除済みな)ことを理由に
再度追加してしまい、削除操作が効かないというバグになった。

## なぜ重要か

「読み込み時に移行する」ロジックは書きやすく一見正しく見えるが、
**移行元のデータを消さない限り、移行は毎回やり直される**。
特に「リストに無ければ追加」のような冪等化条件だけでは、
「ユーザーが意図的に削除した」状態と「まだ移行されていない」状態を区別できない。

## 対策

移行ロジックを書いたら、移行後の状態を保存する経路(このケースでは `configSave()`)で
旧キーを明示的に削除する(`Preferences::remove()`)。読み込み専用の移行(副作用なしの `configLoad()`)を保ちたい場合は、
書き込みが発生するタイミングで必ず旧キーを掃除する設計にする。

kikimimi では issue 00011(複数 WiFi 対応)で発生。`m5stack-cores3/src/config_store.cpp` の `configSave()` を参照。
