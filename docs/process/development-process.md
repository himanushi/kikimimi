# 開発プロセス(AI 駆動開発の運用方針)

調査結果(`docs/analysis/` 3 本)と前作 yesman での運用実績をもとにした、このプロジェクトの開発手順。

---

## 原則: 人間の指示が少なくて済む 3 条件

| 指示が発生する原因 | 対策 |
|---|---|
| 検証を人に頼る | **自己検証ループ**: ビルド・テスト・実機/エミュレータでの動作確認まで AI が自分で回す |
| 決定が文書化されていない | **repo が唯一の記憶**: 決定は ADR(`docs/decisions/`)に追記 |
| 次にやることが不明 | **ROADMAP 駆動**: `docs/ROADMAP.md` のチェックリストを上から消化 |

## ワークフロー(1 タスクの流れ)

```
1. Spec/Issue 化   … タスクを issues/NNNNN/plan.md に切る(What/Why → How の順)
2. 計画承認        … 大きい変更は計画を人間が確認してから実装(小修正は省略可)
3. 実装            … サブエージェントまたはメインが実装。TDD(red → green)
4. 自己検証        … typecheck / test / ビルド / 実機確認。成功は証拠(出力)で示す
5. レビュー        … フレッシュコンテキストの AI レビュー + 人間が設計・意図を確認
6. 記録            … ROADMAP 更新、決定は ADR、ハマりは knowhow/ へ
7. コミット        … 日本語メッセージ、1 スライス = 1 コミット。常に動く状態で終える
```

- タスクの大きさで使い分ける: **diff を一文で説明できる修正は 1→2 を省略**。大きめの機能は仕様(受け入れ基準つき)を先に書き、承認を挟む
- 独立した issue(触るファイルが交差しない)は並列実行可

## ドキュメント体系

| 場所 | 役割 | 更新タイミング |
|---|---|---|
| `CLAUDE.md` | 不変ルールのみ(禁止事項・規約・コマンド)。短く保つ | ほぼ更新しない |
| `docs/decisions/` | ADR(決定の記録) | 決定のたび追記 |
| `docs/specs/` | 要件・設計 | 機能追加時 |
| `docs/analysis/` | 調査レポート | 調査のたび |
| `docs/ROADMAP.md` | チェックリスト(各項目に完了条件) | スライス完了ごと |
| `knowhow/` | ハマった点を 1 件 1 ファイルで蓄積。実装前に必読 | ハマるたび |
| `issues/NNNNN/` | plan.md(着手前の仕様)/ log.md(進行ログ: どう進めたか・途中の判断の why)/ result.md(検証の証跡) | タスクごと |

運用ループの核: **AI が間違えるたびに CLAUDE.md か knowhow/ に 1 行追記する**(失敗駆動追記)。

## why の置き場所(コードコメントと ADR の使い分け)

what / how はコードが表現する。why はスコープで置き場所を分ける:

| why の種類 | 置き場所 |
|---|---|
| 1 行・1 関数に閉じた判断(なぜこの順序でないと壊れるか、なぜこの値か) | コードコメント |
| 1 issue に閉じる判断(この実装でなぜこの方式を選んだか) | `issues/NNNNN/log.md` |
| 複数ファイル・リポジトリ全体に効く判断(ツール選定、構成、方式) | ADR |
| 「やらないこと」の決定、却下した選択肢とその理由 | ADR(コードに現れないため) |
| コードより長生きする判断(コードが書き換わっても残したい経緯) | ADR |

- **コメントはコードが示せない制約・理由だけを書く**。次の行が何をするかの説明、どこから来たか、変更の正当化(レビュー向けの弁明)は書かない。消しても情報が失われないコメントは書かない
- ADR は迷った末に決めたときだけ書く。自明な判断まで記録しない
- ADR は追記専用の履歴。決定を覆すときは既存 ADR を編集せず、新しい ADR で「NNNNN を置換する」と宣言する

## TDD

- テストを先に書く(red → green)。failing test が「偽装できない Done の定義」になる
- ただしファームウェア(M5Stack)はテスト困難な領域が多い。**ロジックは純関数に分離**してホスト側でテスト可能にし、I/O・ハードウェア依存部は実機での動作確認(証拠つき報告)で代替する
- AI にコードとテストを同時に書かせるとトートロジーテスト(実装の写経)になりがち。テストは仕様(受け入れ基準)から導く

## レビューと品質ゲート

- **文脈と強制を区別する**: 守らせたいルールは CLAUDE.md に書くだけでなく、hook・CI(typecheck / test / lint)で機械的に強制する
- レビューは実装者と分離: フレッシュコンテキストのエージェントに diff と基準だけを渡す。指摘は正しさ・要件に影響するものに絞る
- 新規依存パッケージは公式レジストリで実在と正規名を確認する(slopsquatting 対策)
- LLM を使う機能(対話・知識抽出)は、失敗例を貯めてゴールデンデータ化し、eval で回帰検出する(仕組みは必要になってから作る)

## skills / 設定の方針

- skill は反復する手順が固まったものだけ作る(先に作ると手順の方が先に陳腐化する)
- `.claude/settings.json` の permission allowlist で許可プロンプト自体を減らす
- CI(GitHub Actions)で typecheck + test を push 時に実行

## public リポジトリの注意

- 個人情報・メールアドレス・API キー・デバイス固有情報をコード / ドキュメント / コミットメッセージに含めない
- 秘密情報はビルド時の環境変数またはサーバ側 secret で注入する
