# AI コンテキスト管理・知識蓄積の調査

AI コーディングエージェントにプロジェクト知識をどう持たせ、どう蓄積するかの調査。

---

## 1. 大前提: Context Engineering

コンテキストは有限で、トークンが増えるほど想起精度が下がる(context rot)。核心原則は「望む結果の確率を最大化する**最小の高シグナルトークン集合**」を探すこと。

- システムプロンプト/ルールは「適切な高度」で書く — 脆いハードコードと曖昧な指針の中間。具体的だが柔軟なヒューリスティック
- 例示は網羅的なエッジケース列挙ではなく、多様で正準的な少数例を厳選
- 出典: https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents

## 2. プロジェクトメモリファイル(CLAUDE.md / AGENTS.md)

### 書き方のコツ

- 命令形の直接的な指示で書く。否定ルール(「〜をコミットするな」)も肯定ルールと同等に重要
- **短く保つ**(目安 150〜200 行未満)。長すぎると重要ルールがノイズに埋もれる。エージェントが指示なしでも正しくやることは削除
- コードから容易に推論できる情報は書かない。「AI の判断を実際に変える情報」だけ
- ファイル構造は書かない(移動で腐る)。コマンドはコピペ実行可能な形で書く
- 複数ステップの手順は skill へ、パス限定ルールは path-scoped rule へ、強制したい禁止事項は hook へ移す(メモリファイルは文脈であり強制ではない)
- **LLM に自動生成させない** — 研究では LLM 生成の AGENTS.md はタスク成功率をむしろ下げる(一般論でトークンを浪費)。教えなければ分からない固有の規約のみ人が書く
- 段階的開示: 全部詰め込まず、詳細ドキュメントへのポインタを置く
- 出典: https://code.claude.com/docs/en/memory / https://agents.md/

### 運用ループ

**エージェントが間違えるたびにメモリファイルに 1 行追記する**のが知識蓄積の基本ループ。

## 3. モノレポでのルール配置

1. ネスト型: ルートに全体規約、各パッケージに個別の CLAUDE.md(サブディレクトリのものはそのツリーのファイルを読んだとき遅延ロード)
2. path-scoped rules(`.claude/rules/` + `paths:` glob): ファイル種別単位でルール適用
3. 注意: path-scoped rule はファイル「読み込み」時に発火し、新規ファイル作成時には効かない — 作成時の規約はルートに置く

出典: https://code.claude.com/docs/en/large-codebases

## 4. ADR と knowhow の蓄積

### ADR(Architecture Decision Record)

- 「タイトル・文脈・決定・帰結」形式の意思決定記録。エージェントが数分でコードを生成する時代、根拠が生成コードに埋もれて消えるため「なぜこの設計か」を AI が読める形で残す
- メモリファイルから ADR を参照させ「アーキテクチャ判断の前に確認し、矛盾しないこと。未カバーの判断は新規 ADR を先に書くこと」と指示する
- 注意: コンテキストに載せる=強制ではない。重要な決定は hook / CI での機械的チェックに落とす

### lessons-learned(knowhow)

- ハマった点・失敗を 1 件 1 ファイルで蓄積し、実装前に必読させる(並列エージェントの書き込み衝突も回避できる)
- skill で最も価値があるのは実際の失敗に基づく Gotchas セクションで、継続更新するもの(Anthropic エンジニア談)

## 5. AI が読みやすいドキュメント構造

- **自己完結ページ**: エージェントは複数ページの情報を確実に縫合できない。1 トピックは 1 セクションに完結させる
- 階層的見出し(H2 単位で部分回答を抽出できる構造)
- Markdown > HTML(同内容で約 1/6 のトークン)
- 実行可能性重視: 「クリックせよ」ではなく実行可能コマンドで書く

## 6. Skills / Subagents の使い分け

| 手段 | 役割 |
|---|---|
| CLAUDE.md / AGENTS.md | 常時オンの文脈(短く) |
| path-scoped rules | パス限定のハード制約 |
| skills | 再利用する複数ステップ手順(必要時に動的ロード=段階的開示) |
| subagents | コンテキストを汚したくない委譲(調査・レビュー) |
| hooks | 決定論的な強制(lint・テスト・禁止事項) |

- skill は反復する手順が固まったものだけ作る(先に作ると手順の方が先に陳腐化する)
- レビューは「diff と基準だけを見る」フレッシュコンテキストの subagent に(変更を生んだ推論を知らないので客観的)
- 出典: https://claude.com/blog/steering-claude-code-skills-hooks-rules-subagents-and-more / https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills

## 7. まとめ

1. コンテキストは希少資源 — 短く・高シグナルに・段階的開示で
2. 知識蓄積は「失敗駆動追記」— 間違えるたびに 1 行足すループが最強
3. 文脈と強制を区別 — メモリファイルは助言、hook/CI は強制
4. 層の使い分け — 常時オン → パススコープ → オンデマンド → 隔離
5. 手書き優先 — メモリファイルの自動生成は成功率を下げる
