# AI 駆動開発のワークフロー調査

AI にコードを書かせる開発プロセスのベストプラクティス調査。一次情報(Anthropic / GitHub / Cursor / Kiro の公式ドキュメント、著名エンジニアの記事)ベース。

---

## 1. Spec-Driven Development(仕様駆動開発)

### 何か

AI にコードを書かせる前に、レビュー可能な「仕様書」という明示的な成果物を生成・承認するワークフロー。プロンプトと違い、仕様は編集・共有・監査ができる。仕様を「書き捨てのプロンプト」ではなく「継続参照される中心成果物」に格上げする。

### GitHub Spec Kit

- フロー: constitution(プロジェクト憲法)→ specify(What/Why のみ、技術スタックは書かない)→ plan(技術実装戦略)→ tasks(依存関係つきタスク分解)→ implement
- 補助: clarify(曖昧箇所の明確化を plan 前に実施)、analyze(実装前の成果物間整合性チェック)
- 各フェーズで人間の明示的承認を必須にし、暴走を防ぐ
- 出典: https://github.com/github/spec-kit / https://github.github.com/spec-kit/

### Kiro

- 3 成果物: `requirements.md`(ユーザーストーリー+受け入れ基準)/ `design.md`(アーキテクチャ)/ `tasks.md`(実装タスク)
- 仕様変更を検知して実装計画を再調整する「生きた仕様」
- 出典: https://kiro.dev/docs/specs/

### 使い所の限界

Martin Fowler サイトの検証では、小さなバグ修正に full SDD を使うのは過剰(4 ストーリー・16 受け入れ基準に膨張した例)。大きめの機能・グリーンフィールド向き。
出典: https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html

## 2. Plan-then-Implement(計画→実装の分離)

いきなりコーディングさせると「間違った問題を解く」コードが出るため、計画段階で人間が方向修正できるようにする。現在ほぼ全ツール共通のコンセンサス。

- Claude Code 公式手順: Explore(plan mode で現状理解のみ)→ Plan(計画を人間が編集)→ Implement(実装+テスト)→ Commit/PR
- ただし「diff を一文で説明できるタスクなら plan を省略せよ」と公式が明記
- 発展形(インタビュー駆動仕様): 大きめの機能では AI に質問させてエッジケース・トレードオフを掘り下げて SPEC.md を書かせ、**新しいセッション**のクリーンなコンテキストで実装する
- 実践プロンプト例: 「計画を出したら止まれ。私が approved と言うまでコードを書くな」
- 出典: https://code.claude.com/docs/en/best-practices / https://cursor.com/blog/agent-best-practices / https://addyosmani.com/blog/ai-coding-workflow/

## 3. Issue 駆動開発

GitHub Issue(またはリポジトリ内の issue ファイル)を「エージェントへの制御面」として使う。Issue は意図の記述・議論・履歴保存を備えた成果物であり、コード生成より難しい「制御」の問題を解く。

- Issue → エージェントがタスクごとに新ブランチで作業 → PR 提出、ベースブランチへの直接 push は不可
- ガードレール(GitHub 公式): デフォルト read-only 権限、PR の自動マージ禁止・人間承認必須、プロンプトインジェクション対策のサンドボックス
- 出典: https://github.blog/ai-and-ml/automate-repository-tasks-with-github-agentic-workflows/

## 4. エージェント運用(コンテキスト・並列・ガードレール)

### コンテキスト管理が最重要制約

- LLM 性能はコンテキストが埋まるほど劣化する。無関係タスク間で `/clear` 必須
- 「2 回修正しても直らなければクリアして、学んだことを盛り込んだ良いプロンプトで再開」— 修正を蓄積した長いセッションより、良いプロンプトのクリーンなセッションがほぼ常に勝つ
- CLAUDE.md は短く保つ(「この行を消したらミスするか?」で剪定)

### Subagent の使い方

- 調査の隔離: 多数のファイルを読む調査は subagent に委譲し、メインの実装コンテキストを汚さない
- 検証の分離: 実装したエージェント自身に採点させず、フレッシュコンテキストの subagent に diff と基準だけを渡してレビューさせる。ただし「レビュアーは頼まれれば必ず何か指摘する」ので、正しさ・要件に影響するギャップのみ報告させる

### 並列実行

- git worktree で各セッションを隔離して並列開発(Claude Code / Cursor とも公式推奨)
- Writer/Reviewer パターン: セッション A が実装、セッション B(フレッシュ)がレビュー
- 大規模移行の fan-out は 2〜3 ファイルで試してプロンプトを修正してから全量実行

### 自動化ガードレール

- CLAUDE.md の指示は「助言」、hooks は「決定論的」。毎回必ず実行すべきこと(lint・テスト・書き込み禁止)は hook にする
- 検証可能なゴール(テスト・ビルド・スクリーンショット比較)を与えることが「見守るセッション」と「離席できるセッション」の差
- 成功は主張でなく証拠(テスト出力・実行結果)で示させる

## 5. TDD × AI

- コミット済みの failing test は「エージェントが偽装できない Done の定義」。エージェントは明確な反復ターゲットがあるとき最も性能が高い
- 明示的な強制がないと、多くのエージェントは Red フェーズを飛ばし「現状を追認するテスト」を後付けする。hook で強制する手がある
- 反証研究(TDAD): 手続き的な TDD 指示を文脈なしで与えるとむしろリグレッションが増加。影響範囲の文脈情報を与える方が効く。「手順の強制」より「文脈の提供」
- 出典: https://arxiv.org/pdf/2603.17973 / https://www.brgr.one/blog/ai-coding-agents-tdd-enforcement

## 6. Human-in-the-Loop レビュー体制

レイヤー化が 2026 年のコンセンサス:

1. 床: linter・静的解析・型チェックを CI で(人間を書式チェックに使わない)
2. 中間: AI レビュアーが意味的パターンを検査
3. 人間: 設計・意図・プロダクト判断に集中

- AI 生成コードは「正しく見えて微妙に間違う」— エージェントが速いほどレビューが重要
- 400 行未満の小 PR と明確なガイドラインが有効
- Anthropic の安全原則: 最小権限、可逆なアクション優先、高リスク判断には人間承認

## 7. 収束点

**Issue/Spec が意図をスコープ → 計画を人間承認 → エージェントが隔離環境・最小権限で実装 → failing test が客観的完了基準 → hooks/CI が機械的に検証 → フレッシュコンテキストの AI レビュー → 人間が設計・意図をレビューしてマージ**

失敗パターン: 無関係タスク混在セッション、コンテキスト汚染下での修正の繰り返し、肥大化 CLAUDE.md、検証なき信頼、スコープなしの無限探索。
