# 00002: OpenAI API キーはセットアップポータルで登録しデバイスの NVS に保存する

## 文脈

デバイス(CoreS3)が OpenAI Realtime API と直接対話するには API キーが必要だが、public repo・ビルド成果物にキーを含めることはできない(CLAUDE.md)。前作 yesman は「キーはサーバ側 secret、デバイスはサーバ経由」の構成だったが、kikimimi は当面サーバを持たず、デバイス単体で Realtime API に接続する。キーは 164 文字あり、デバイスのソフトキーボードでの手入力は非現実的。

## 決定

初期設定は **SoftAP + Web 設定ポータル**方式にする(issue 00001):

- 未設定時にデバイスが AP(`kikimimi-setup`)を立ち上げ、ブラウザのフォームから WiFi 資格情報と API キーを登録する(コピペ可能)
- AP は WPA2 とし、パスワードは起動ごとの乱数を本体画面にのみ表示する。オープン AP だとフォーム送信時に API キーが無線区間を平文で飛び、電波圏内の第三者に傍受されうるため。フォーム自体は AP 内の平文 HTTP(TLS 化はセルフ署名証明書の警告でセットアップ体験を壊すため見送り。WPA2 で無線区間は暗号化される)
- 登録値は **NVS(Preferences)に平文保存**する。フラッシュ暗号化(ESP32 の flash encryption)は導入しない — 個人宅で使う 1 台のデバイスであり、物理アクセスされる脅威モデルに対してコストが見合わない
- デバイスは NVS のキーで OpenAI へ直接接続する。サーバ中継(ephemeral key 発行)は `server/` を作る段になったら再検討する

却下した選択肢:
- **ビルド時環境変数での埋め込み**: ビルド成果物にキーが残り、書き込みのたびに開発環境が必要。家族など開発者以外がセットアップできない
- **デバイス画面のキーボード入力**(yesman の WiFi 設定方式): 164 文字の無理な手入力になる
- **サーバ中継を最初から作る**: 「単純な対話に徹する」段階でサーバ運用コストと障害点を増やす

## 帰結

- キーの実体はユーザーの手元(OpenAI ダッシュボード)とデバイス NVS のみに存在し、repo・ビルド成果物には一切含まれない
- デバイスを手放す・譲渡する際は NVS 消去(`esptool erase_flash` または設定画面からのリセット機能を将来追加)とキーのローテーションが必要
- ポータルの設定用 Web サーバは、将来のチャット履歴表示・再設定 UI の足場になる
