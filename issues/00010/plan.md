# 00010: 設定ページの STA 常設(AP 切替なしで再設定)

## What / Why

現在、設定変更は AP モードのセットアップポータル経由のみで、再設定のたびにスマホの WiFi を M5Stack の AP に切り替える必要がある。WiFi 接続済み(STA)のときは同じ Web 設定ページを STA 側の IP で常設し、ホーム画面の設定ボタンで URL QR を表示するだけにする。スマホは WiFi を切り替えず QR 1 回で設定ページを開ける。

- WiFi 接続済み: 設定ボタン → 画面に `http://<STA の IP>/` の QR + URL 表示。Web サーバは STA 側で稼働
- WiFi 未設定・接続失敗: 従来どおり AP モードのポータルにフォールバック(変更なし)

依存: 00008(ホーム画面の設定ボタン)。

## 受け入れ基準

- [ ] 「どの設定画面を出すか(STA の URL 案内 / AP ポータル)」の判定が純関数に分離されテストされる(検証: `uvx --from platformio pio test -e native` green。先に red を確認)
- [ ] WiFi 接続済み状態で設定ボタンをタップすると、AP に切り替わらず STA の IP の URL と QR が表示される(検証: 実機。スマホの WiFi 接続先が変わらないまま設定ページが開ける)
- [ ] STA 側の設定ページで SSID / パスワード / API キーを保存でき、保存後は再起動して新設定で接続する(検証: 実機)
- [ ] QR 表示画面から戻る操作がある(タップで閉じる等)(検証: 実機)
- [ ] WiFi 未設定・接続失敗時は従来の AP ポータルが起動する(検証: 既存挙動の維持。実機)
- [ ] cores3 ビルドが通る(検証: `uvx --from platformio pio run -e cores3` SUCCESS)

## やらないこと

- 設定ページの見た目の大幅な変更(フォーム構成は現行を流用)
- 複数 WiFi の登録 → 00011
- mDNS(`kikimimi.local`)での名前解決(必要になったら別 issue)

## How(実装方針)

- `src/portal.*`: WebServer の起動を AP モードから分離し、STA モードでも同じハンドラ群で起動できるようにする(captive 302 は AP のときだけ)
- `src/main.cpp`: 設定ボタン → STA なら「設定 URL QR 画面」を表示(既存 drawUrlGuideScreen 相当を流用)。タップで閉じてホームに戻る
- 判定純関数は `src/pure/` に置く(例: settings_entry.*)
- セキュリティ上の注意: 設定ページは同一 LAN 内からアクセス可能になる。API キーの現在値はフォームに表示しない(既存方針の維持)
