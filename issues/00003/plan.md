# 00003: リアルタイム音声対話(マイク送信・音声再生)

## What / Why

00002 のセッション上で双方向の音声を流し、「画面をタップしたら AI と声で会話できる」を完成させる。マイク音声を `input_audio_buffer.append` でストリーミング送信し、`response.output_audio.delta` の音声をスピーカーで再生する。発話の切れ目はサーバ側 VAD(semantic_vad / server_vad)に任せ、push-to-talk にしない。

これがプロジェクトの中核体験(相棒との対話)の最小形。

## 受け入れ基準

- [ ] 音声チャンクの組み立て・分解(PCM16 ↔ base64、append イベント生成)が純関数で分離され、ホスト側テストが通る(検証: `pio test -e native`)
- [ ] ビルドが通る(検証: `pio run -e cores3` の出力)
- [ ] タップで会話開始 → 話しかけると音声で応答が返る(検証: 実機で対話し、往復レイテンシと内容を観察)
- [ ] 応答再生中に次の発話ができる、または再生終了後に続けて話せる(連続対話。検証: 実機で 3 往復以上)
- [ ] 再タップで会話終了し、待機状態に戻る(検証: 実機観察)
- [ ] 会話中の状態(聞いている / 考えている / 話している)が画面で分かる(検証: 実機観察)

## やらないこと

- 会話履歴の保存・知識蓄積(別 issue。サーバ側 `server/` の設計が先)
- 割り込み発話(barge-in)の作り込み(M5Unified は Mic/Speaker 排他のため、まずは交互対話でよい)
- ウェイクワード・常時待機
- 音声品質のチューニング(エコー・ノイズ対策は動いてから)

## How(実装方針)

- 依存: issue 00002(WebSocket セッション)
- 音声フォーマットは PCM16。サンプルレートは Realtime API 標準の 24kHz を M5Unified の Mic/Speaker 設定で合わせる(帯域が厳しければ g711 8kHz に落とす選択肢を検証)
- M5Unified は Mic と Speaker が排他 → 録音フェーズと再生フェーズを明示的に切り替えるステートマシン(yesman の voice.cpp の排他制御ノウハウを流用)
- 送信は 100ms 単位のチャンクを base64 化して append。受信 delta は PSRAM のリングバッファに貯めて再生
- turn detection は `session.update` で semantic_vad を指定(server_vad と比較して選ぶ)
