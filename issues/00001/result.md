## 判断と経緯

- **AP は WPA2(起動ごとの乱数パスワードを本体画面に表示)**: 当初オープン AP だったが、レビュー指摘(下記 2)を受けて変更。固定パスワードは public repo に載ってしまうため、起動ごとに乱数生成し画面にのみ表示する
- **パスワードはトリムしない**: SSID・API キーはコピペ由来の前後空白を除去するが、WiFi パスワードは先頭・末尾の空白も有効な文字のため原文のまま
- **API キーの形式チェックは緩め**(`sk-` プレフィックス + 長さ 20〜512 + 印字可能 ASCII): OpenAI のキー形式は変わりうるので、真の検証は 00002 の接続時に API 側へ任せる
- **captive portal は DNS ワイルドカード + 302 リダイレクト**のみ: iOS/Android の検出 URL への個別対応は動かして問題が出てから

## レビュー指摘と対応

フレッシュコンテキストの subagent レビュー 2 本(片方はミューテーションテストで「実装を 3 通りに壊すと全テストが落ちる」ことまで確認)。指摘と対応:

1. **NVS 保存の fail-open**(medium): `configSave` が `begin()`/`putString` の失敗を握りつぶし、無条件に「保存しました」を返していた → 戻り値を bool 化し、失敗時はエラーページ(500)を返して再起動しない
2. **セットアップ無線区間の API キー平文露出**(low): オープン AP のため POST 中のキーが傍受可能だった → 起動ごとの乱数パスワードで WPA2 化 + ADR 00002 に無線区間の扱いを追記
3. **境界値テストの受理側が片側落ち**(low): パスワード 8 文字ちょうど・API キー上限 512 ちょうどの受理を追加(off-by-one 回帰の捕捉)
4. **検証エラー時に API キーを再表示しない**(low)→ **対応せず(意図的)**: 秘密情報をレスポンス HTML にエコーしない方針を優先。エラーはほぼ SSID/パスワード起因で、キーの貼り直しコストは低い
