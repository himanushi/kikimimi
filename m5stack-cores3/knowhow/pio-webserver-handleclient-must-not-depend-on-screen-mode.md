# WebServer::handleClient() は画面モードに紐付けて呼ばない

- **症状**: STA 常設(issue 00010)で、Web サーバの起動とリクエスト処理(`webServer.handleClient()`)を「URL 案内 QR 画面を表示している間だけ」呼ぶ実装にしたところ、QR 画面をタップで閉じた瞬間にリクエスト処理が止まり、スマホ側でフォーム送信中でも `/save` の POST がハングする不具合になった(コードレビューで発覚。実機では気づきにくい: QR を見ている間だけ試すと問題が出ない)
- **原因**: `loop()` 内で `if (mode == X) webServer.handleClient();` のように、Web サーバの生存期間を特定の画面モードの真偽に結びつけていた。ユーザー操作(タップで画面遷移)と Web サーバのライフサイクルが独立であるべきなのに、同じ変数(`mode`)で管理してしまった
- **対処**: Web サーバを起動したら、それ以降は「稼働中かどうか」を画面モードとは別の条件で判定し、毎ループ確実に `handleClient()` を呼ぶ。kikimimi では STA 用 Web サーバを起動時(WiFi 接続成功直後)に一度だけ起動し、`mode != Mode::PORTAL`(AP モードではない全画面)の間は常に `portalLoop()` を呼ぶ設計にした。QR 画面固有の役割は「案内表示 + タップで戻る」だけに縮小した
- **出典**: issue 00010
